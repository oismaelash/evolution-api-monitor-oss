Criar monitorador de numero da evolution api.

cada pessoa pode ter varios projetos.

cada projeto, tem varios numeros.

a ideia é o gerenciador, verificar a saude de cada numero.

a cada 5 minutos envia um ping (statu e presence),
caso os dois retorne ok, segue a vida.

Se em 3 pings, retorna falha, tentar fazer restart do numero.
faz o restart, espera 1 minuto, e verifica novamente com ping, se a saude esta de volta.

caso nao, tenta fazer mais um restart, apos 1 minuto faz outro ping.

apos segunda tentativa falha, envia um WhatsApp, para o numero da pessoa que gerencia aquele projeto/conta.

falando qual o numero precisa ser reconectado de forma manual, envia tambem, o qrcode de conexao.

envia tambem o numero do pairing code de conexao.

tudo deve ter log, que a pessoa consiga filtrar os log por numero, ou projeto.

apos trazer os numeros, quais numeros quer monitorar

o tempo dos pings, quantidade de tentativas, delay entre tentativas, exponencial ou fixo.

se deseja escolher um numero para enviar o alerta, e criar o template de alerta, caso ele nao use o Monitor, fallback de numero para enviar mensagem, sendo que por padrao, caso nao escolha o Monitor, o Monitor é o ultimo fallback

Normalização de falhas

Nem toda falha é igual.

Você precisa classificar:

NETWORK_ERROR
AUTH_ERROR
INSTANCE_NOT_FOUND
RATE_LIMIT
UNKNOWN

👉 Isso melhora:

retry inteligente
alerta mais útil

Anti-spam de alertas

Imagina isso:

número falha → alerta enviado
continua falhando → spam infinito

👉 Solução:

cooldown de alerta (ex: 30 min)


Vitest para testes

iremos rodar em uma vps o monolito (api e frontend) e o worker

Mas precisa definir as transições, senão o worker vira caos.

Exemplo:

CONNECTED → (falha ping) → DISCONNECTED  
DISCONNECTED → (retry) → CONNECTING  
CONNECTING → (sucesso) → CONNECTED  
CONNECTING → (falha) → ERROR  
ERROR → (restart) → RESTARTING  
RESTARTING → (sucesso) → CONNECTED  
RESTARTING → (falha) → ERROR
